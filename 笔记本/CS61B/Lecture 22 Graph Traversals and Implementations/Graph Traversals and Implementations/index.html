    Shortest Paths¶

    1. 回顾：DFS vs BFS 找路径¶

    之前学过的两种图遍历方式，在**找路径**这个问题上的对比：

    考量 DFS BFS
    正确性 ✓ 都能找到路径 ✓ 都能找到路径
    输出质量 随便一条路径 最短路径（边数最少），2合1 deal
    时间效率 \(O(V+E)\)，两者差不多 \(O(V+E)\)
    空间效率 深瘦图差（调用栈很深） 宽胖图差（队列会很大）

    为什么 DFS 输出的是随便一条路径？

    DFS 的策略是"一条路走到底，走不通再回头"。它 不关心层级 ，碰到第一条通往终点的路就直接返回了。

    举个例子： 

    1
2
3
    A — B — D — F — T
|               |
C — E — G — H — I
    从 A 出发找去 T 的路：DFS 会先走 A→B→D→F→T，一看到 T 就停下来了，直接返回这条路径。但实际上 A→C→E→G→H→I→T 可能更短，DFS 根本不会去看那条路。

    如果运气不好，DFS 可能先绕了一大圈才到终点。

    为什么 BFS 输出的是最短路径（边数最少）？

    BFS 的核心是 按层级一层一层往外扩散 ：

    • 第 0 层：只有源点 \(s\)
    • 第 1 层：所有与 \(s\) 直接相连的邻居（距离 = 1 条边）
    • 第 2 层：所有需要 2 条边才能到达的点
    • 第 \(k\) 层：所有需要 \(k\) 条边才能到达的点

    这意味着：BFS 第一次碰到目标节点时，一定是最少的边数。因为如果存在一条更短的路径，那个节点早就被发现了。

    用刚才的图验证： - BFS 第 1 层：B, C - BFS 第 2 层：D, E - BFS 第 3 层：F, G - BFS 第 4 层：H - BFS 第 5 层：I - BFS 第 6 层：T

    BFS 会按层级走完，找到最短的那条路径。而 DFS 直接冲上去就完事了。

    空间效率对比：为什么深瘦图 DFS 差，宽胖图 BFS 差？

    这个问题的核心在于：DFS 和 BFS 分别需要记住哪些信息。

    DFS 用栈（递归调用栈）： 只需要记住"当前路径"上还没探索完的分支。

    在深瘦图（比如一条长链）中，DFS 的调用栈会非常深： 

    1
2
    深瘦图：A → B → C → D → E → F → G → ... → Z
DFS 的栈：[A, B, C, D, E, F, G, ..., Z]  ← 栈深 = 节点总数
    最坏情况：\(O(V)\) 的栈空间。图有 \(V\) 个顶点就栈深 \(V\)。

    但在宽胖图（每个节点有很多邻居）中，DFS 反而还行：因为一条路很快就走到头了，栈不深。

    BFS 用队列： 需要同时记住"当前层"所有还没处理的节点。

    在宽胖图（比如一颗大树或者星形图）中，某一层的节点数可以非常大： 

    1
2
    宽胖图：中心节点连接 1000 个邻居
BFS 的队列：[邻居1, 邻居2, ..., 邻居1000]  ← 队列长度 = 1000
    最坏情况：\(O(V)\) 的队列空间（星形图，所有节点都在第 1 层）。

    但在深瘦图（长链）中，BFS 反而还行：因为每层只有 1 个节点，队列最多也就 1 个元素。

    总结： - DFS 的空间取决于**最大深度**（最深的路径有多长） - BFS 的空间取决于**最大宽度**（最宽的一层有多少节点）

    BFS 用于地图导航的问题

    BFS 返回的是 边数最少 的路径，但在实际地图中，我们关心的是 总距离最短 。

    比如从起点 \(s\) 到终点 \(t\)，BFS 可能选了一条有 3 条边但总距离 190 的路，而实际最优路径虽然边数多但总距离只有 110。

    核心问题： BFS 没有考虑边的权重（edge weights）！

    2. Dijkstra's Algorithm¶

    2.1 从一个问题入手¶

    Single Source Single Target Shortest Paths： 给定一个带权图，找从源点 \(s\) 到目标 \(t\) 的最短路径。

    Single Source Shortest Paths： 找从 \(s\) 到 每个可达顶点 的最短路径。

    示例

    找从镇 0 到镇 5 的最短路径：

    • 最佳路径：\(0 \to 2 \to 4 \to 5\)，总长度 9 英里
    • 注意：路径 \(0 \to 2 \to 5\) 虽然只经过 3 个镇，但总长度是 16 英里，不是最短的

    观察：最短路径一定 没有环（假设边权重非负）。

    2.2 Shortest Paths Tree (SPT)¶

    对于 Single Source Shortest Paths 问题，解的结构是一棵树：

    • 如果图有 \(V\) 个顶点且连通，SPT 有 \(V-1\) 条边。
    • 每个顶点（除了源点）恰好有一条输入边。
    • 可以看作是从 \(s\) 到所有顶点的最短路径的并集。

    为什么解的结构是一棵树？

    两个关键性质保证了 SPT 是树而不是别的什么结构：

    性质 1：每个顶点只有一个父节点。

    算法中 edgeTo[v] 记录的是到达 \(v\) 的最短路径上的前驱节点。每次松弛成功时，edgeTo[v] 被更新为新的前驱。最终稳定后，每个顶点 \(v\) 恰好只有一个 edgeTo[v]——它只有一个"父亲"。

    性质 2：不会有环。

    因为边权重非负，最短路径**不可能走回头路**。假设 SPT 中有一个环：\(A \to B \to C \to A\)，那意味着 \(A\) 的父亲是 \(C\)，\(C\) 的父亲是 \(B\)，\(B\) 的父亲是 \(A\)。但这样 \(A\) 到自己的距离就是"走一圈再回来"，必然 ≥ 0，而最短路径应该直接停在 0。矛盾。

    所以 SPT 无环 + 每个节点一个父节点 + 连通 = 树。

    直观理解：

    从源点 \(s\) 出发，每发现一个新顶点，就用一条边把它"连"到已知的最短路径上。\(V\) 个顶点的连通图，用 \(V-1\) 条边连起来，没有环，就是一棵树。

    1
2
3
4
5
6
7
8
9
    原图可能有环：          SPT（从 A 出发）：

    A ── B                 A
   /│\   │               / | \
  C │ D──E              B  C  D
   \|/                  |
    F                   E
                         \
                          F

    SPT 只保留"最短路"上需要的边，把多余的边（形成环的那些）全丢弃了。

    2.3 核心概念：Edge Relaxation（边松弛）¶

    这个算法在错误算法2与Dijkstra算法中都用到了，虽然错误算法2用到了它也还是错误的算法，但是边松弛算法是Dijkstra算法的灵魂。

    松弛一条边 \(p \to q\)（权重为 \(w\)）：

    1
2
3
    如果 distTo[p] + w < distTo[q]:
    distTo[q] = distTo[p] + w
    edgeTo[q] = p

    意思就是：如果通过 \(p\) 走到 \(q\) 比目前已知的到 \(q\) 的路径更短，就 更新 \(q\) 的距离和前驱。

    手动模拟找 SPT

    以源点 A 为例，手动标记每个点到 A 的最短距离：

    1
2
3
4
5
        A──(5)──B
    │       │
   (1)     (2)
    │       │
    C──(5)──D

    从 A 出发： - A 到 A：0 - A 到 C：1（直接走 A→C） - A 到 B：2（走 A→C→B，而不是直接 A→B 的 5） - A 到 D：4（走 A→C→B→D）

    所以 SPT 的边是：A→C, C→B, B→D。

    2.4 从错误算法到正确算法¶

    错误算法1：直接 DFS¶

    按 DFS 顺序加边到 SPT 里，不管距离。

    结果：可能选到很长的路径。比如 A→B 直接加了 5 的距离，但其实走 A→C→B 只需要 2。

    错误算法2：DFS + 松弛¶

    DFS 遍历，但遇到更短的路径时就更新（松弛）。

    问题：DFS 的遍历顺序是"深度优先"，可能先把远处的节点"定死"了，后来发现近的路径，导致一串节点都要重新更新。

    例子：为什么"来不及回头"？

    1
2
3
4
5
6
7
8
9
               A
        /     \
      10         1
      /           \
     B ───(1)──── C
      \           \
       1           1
        \          \
         D ──(1)── E ──(1)── F

    DFS 走的路线：A → B → C → E → F，回溯到 C，回溯到 B → D，回溯到 A

    1. A→B（10）：dist[B] = 10
    2. B→C（1）：dist[C] = 11
    3. C→E（1）：dist[E] = 12
    4. E→F（1）：dist[F] = 13
    5. F 没有未访问邻居，回溯到 E，回溯到 C，回溯到 B
    6. B→D（1）：dist[D] = 11
    7. D 没有未访问邻居，回溯到 B，回溯到 A
    8. A→C（1）：1 < 11，更新 dist[C] = 1

    问题来了： dist[C] 从 11 降到了 1。但 E 和 F 早就被 DFS 走完了： - E 实际最短应该是 dist[C] + 1 = 2，但现在是 12 - F 实际最短应该是 dist[E] + 1 = 3，但现在是 13

    这就是"牵一发而动全身"： C 的距离一变，E 要跟着变；E 一变，F 也要跟着变——像多米诺骨牌一样，一层一层往回推。但 DFS 已经回溯过了 E 和 F，不会再回头去重新松弛它们。

    你可能会想："那发现 C 变短的时候，顺便把 E 和 F 也更新了不就行了？"但问题是，E 和 F 下游可能还有更多节点，都要跟着更新。而且如果 E 还有其他入边，情况会更复杂——你无法确定哪些节点需要重新算，只能全部重新跑一遍。DFS 的顺序决定了它"一锤子买卖"——先走到哪算哪，回头更新的代价太高。

    正确的做法（Dijkstra）： 按距离从小到大处理节点，确保每个节点第一次被"定死"时，就是最终的最短距离，不需要回头。

    // 初始化
PQ.add(source, 0)
for (其他顶点 v) PQ.add(v, infinity)

// 主循环
while (PQ 不为空) {
    p = PQ.removeSmallest()  // 取出最近的顶点
    // 松弛 p 的所有出边
    for (每条边 p→q，权重 w) {
        if (distTo[p] + w < distTo[q]) {
            distTo[q] = distTo[p] + w
            edgeTo[q] = p
            PQ.changePriority(q, distTo[q])
        }
    }
}

    关键不变量： - edgeTo[v] 是 v 的当前最优前驱（离V最近的点） - distTo[v] 是已知的从源点到 v 的最优距离 - PQ(PriorityQueue) 按 distTo 排序存储所有未访问的顶点

    重要性质： - 总是按从源点的 总距离从小到大 访问顶点 - 对已访问（白色）顶点的松弛**总是失败** （不会再更新）

    3. Dijkstra 的正确性与运行时间¶

    3.1 为什么 Dijkstra 是正确的？¶

    定理： 如果所有边权重非负，Dijkstra 保证返回正确结果。

    证明思路（归纳法）： 1. distTo[source] = 0，显然最优。 2. 放松完源点(source)的所有边后，源点就变白了，设 \(v_1\) 是 distTo 最小的顶点（即离源点最近的顶点）。 3. 此时我们Claim：distTo[v1] 已经是最优的，之后任何对 \(v_1\) 的松弛都会失败。 为什么？ - 对于任何还没出队的顶点 \(p\)，有 distTo[p] ≥ distTo[v1] - 因此 distTo[p] + w ≥ distTo[v1]（因为 \(w \ge 0\)） - 所以不可能通过任何未出队节点找到比 \(v_1\) 更短的路径

    可以用数学归纳法证明：每个顶点出队后，上述性质对所有顶点都成立。

    3.2 负权重边的问题¶

    Dijkstra 在有负权重边时会失败！

    原因： 已访问（白色）顶点的松弛**可能成功**。

    例子： 

    1
2
    顶点 33 → 34（权重 1）
顶点 34 → 14（权重 -6）
    即使 34 出队时的 distTo 比 14 大，它仍然可以通过负权重边 -6 来更新已经访问过的顶点 14 的 distTo。这破坏了"已访问顶点的 distTo 不再改变"这个关键性质。

    所以 Dijkstra 不能处理负权重边。

    3.3 时间复杂度¶

    假设使用 二叉堆 实现的优先队列：

    PQ 操作 次数 每次代价 总代价
    add \(V\) \(O(\log V)\) \(O(V \log V)\)
    removeSmallest \(V\) \(O(\log V)\) \(O(V \log V)\)
    changePriority \(E\) \(O(\log V)\) \(O(E \log V)\)

    总时间复杂度： \(O(E \log V)\)（假设 \(E \ge V\)）

    3.3.1 操作次数的解析：¶

    add 为什么是 V 次？

    初始化时，我们把所有 \(V\) 个顶点都加入 PQ。每个顶点只加一次。

    removeSmallest 为什么是 V 次？

    主循环 while (PQ 不为空) 中，每次循环出队一个顶点。\(V\) 个顶点各出队一次，总共 \(V\) 次。

    changePriority 为什么是 E 次？

    这是最关键的一点。看伪代码：

    1
2
3
4
5
6
7
    for (每条边 p→q，权重 w) {
    if (distTo[p] + w < distTo[q]) {
        distTo[q] = distTo[p] + w
        edgeTo[q] = p
        PQ.changePriority(q, distTo[q])  ← 这里
    }
}

    外层循环：每个顶点出队一次（\(V\) 次）。内层循环：遍历该顶点的所有出边。所有顶点的出边加起来就是总边数 \(E\)。 每条边最多触发一次 changePriority（只有松弛成功时才会调用），所以总调用次数 ≤ \(E\)。

    为什么总代价是 \(O(E \log V)\)？

    三项分别是 \(O(V \log V)\)、\(O(V \log V)\)、\(O(E \log V)\)。在连通图中 \(E \ge V - 1\)，所以 \(E \log V\) 是主导项，总复杂度 = \(O(E \log V)\)。

    3.3.2 为什么每次 PQ 操作是 \(O(\log V)\)？¶

    这里的优先队列是用 二叉堆（binary heap） 实现的。二叉堆是一棵**完全二叉树**，树中有 \(V\) 个节点。

    树的高度 = \(\log_2 V\)

    一棵有 \(V\) 个节点的完全二叉树，高度是 \(\lfloor \log_2 V \rfloor\)。例如 \(V = 16\) 时，树高 = 4；\(V = 1000\) 时，树高 ≈ 10。

    每次 PQ 操作的代价就是树的高度：

    操作 堆内发生了什么 最多交换次数
    add 新元素放到堆底，然后 swim（上浮）到合适位置 从最底层浮到根，最多 \(\log V\) 次交换
    removeSmallest 把堆顶（最小值）拿走，堆底元素补到堆顶，然后 sink（下沉）到合适位置 从根沉到最底层，最多 \(\log V\) 次交换
    changePriority 某个节点的优先级变了，可能需要 swim（优先级变高）或 sink（优先级变低） 最多 \(\log V\) 次交换

    swim 和 sink 的直观理解：

    1
2
3
4
5
6
7
    swim（上浮）：                    sink（下沉）：
      ○                                ○
     / \                              / \
    ○  ○                            ○   ○
   / \                              / \
  ○  ○  ← 新元素，比父节点小        ○    ○  ← 堆底补上来的大元素
                                    比两个孩子都大，往下沉

    swim：和父节点比较，比自己小就交换，一路往上。sink：和两个孩子比较，比自己大就交换，一路往下。无论哪种情况，最多从树底到树顶，步数 = 树高 = \(\log V\)。

    所以每次 PQ 操作是 \(O(\log V)\)，\(E\) 次 changePriority 就是 \(O(E \log V)\)。

    4. A* 算法（CS188中的，这里了解一下）¶

    4.1 Dijkstra 用于导航的问题¶

    Dijkstra 会 均匀地向四周扩散 。如果用来做地图导航（比如从 Denver 到 NYC），它会探索 Denver 周围几乎所有地方，包括完全反方向的区域。

    但是我们要找的仅仅是到达NYC这个目标的最短距离，不需要找到达其它点的最短距离，因此用Dijkstra算法会非常浪费时间。

    为了解决这个问题，我们引入了A*算法（CS188中的）

    4.2 A* 的核心思想¶

    简单想法： 访问顶点的优先级不是只看 d(source, v)，而是看：

    \[\text{priority} = d(source, v) + h(v, goal)\]

    其中 \(h(v, goal)\) 是一个 启发函数（heuristic function） ，用来 估计 从 \(v\) 到目标 \(goal\) 还要走多远。

    什么叫"估计"？h 到底是什么？

    \(h\) 可以是 任意的 ，但通常取一个"有道理的猜测"。PPT 里的例子中，下图中 \(h(v, goal)\) 取的是 \(v\) 的所有出边中的最小权重。下图的完整流程点击这里观看。也就是说：假设从 \(v\) 出发，运气最好的情况（每一步都走最小权重的边）到 goal 大概需要多远。

    这个 h 有什么用？ 它给 PQ 排序时多加了一个"朝向目标"的因素：

    • \(d(source, v)\)：已经走了多远
    • \(h(v, goal)\)：**预估**还要走多远

    • 两者相加：这个顶点"整体"离目标有多近

    • 和 Dijkstra 的区别：Dijkstra 只看已经走了多远（\(d(source, v)\)），A* 还加了**预估还要走多远**（\(h(v, goal)\)），所以会优先探索"方向对"的顶点。

    • 例子：Henderson 比 Englewood 离 Denver 更远（\(d\) 更大），但 Henderson 更靠东（\(h\) 更小），加起来可能反而更优先。

    // 松弛 v 的所有出边

            edgeTo[w] = v
            PQ.changePriority(w, distTo[w] + h(w, goal))  // 注意这里加了 h
        }
    }
}

    对于地图导航，可以用 直线距离（as-the-crow-flies distance） 作为启发函数：

    不需要完美！ 启发函数就是个估计，不用精确。

    5. 启发函数的质量（CS188 预览）¶

    5.1 极端情况分析¶

    设置 效果
    \(h(v, goal) = 0\) 退化为 Dijkstra（没有方向性）
    \(h(v, goal) = 10000\)（常数） 还是 Dijkstra（每个点都加同一个数，相对顺序不变）

    5.2 坏的启发函数¶

    启发函数高估会导致错误结果

    如果你"觉得"伊利诺伊州和印第安纳州是荒郊野岭，给它们设 \(h = 100000\)：

    • 算法会**绕开这些州**，即使穿过它们可能是最短路径
    • 结果错误！

    5.3 保证正确性的条件¶

    要让 A* 给出正确结果，启发函数需要满足：

    • Admissible（可接纳）： \(h(v, goal) \le\) 从 \(v\) 到 \(goal\) 的**真实**最短距离。

    • 简单说：永远不要高估。

    • Consistent（一致）： 对于每个邻居 \(w\)： $\(h(v, goal) \le dist(v, w) + h(w, goal)\)$

    • 简单说：从 \(v\) 到目标的估计距离，不超过走到邻居的实际距离加上邻居到目标的估计距离。

    Tip

    这部分是 AI 课程 CS188 的内容，CS61B 不要求掌握。

    你只需要知道：启发函数的选择很重要，选错了 A* 会给出错误答案。

    所有 consistent 的启发函数一定是 admissible 的。

    6. 总结：图问题全景¶

    问题 描述 算法 时间复杂度
    找路径 从 \(s\) 到每个可达顶点的一条路径 DFS \(O(V+E)\)
    无权最短路径 从 \(s\) 到每个可达顶点的最短路径（边数最少） BFS \(O(V+E)\)
    有权最短路径（全目标） 从 \(s\) 到每个顶点的最短加权路径 Dijkstra \(O(E \log V)\)
    有权最短路径（单目标） 从 \(s\) 到某个特定 \(t\) 的最短加权路径 A* 取决于启发函数

    关键洞察： - 当边权重都相等时，Dijkstra 就是 BFS。 - 当 \(h(v, goal) = 0\) 时，A* 就是 Dijkstra。 - A* 通过启发函数提供了 方向性 ，在单目标问题中通常比 Dijkstra 快很多。
