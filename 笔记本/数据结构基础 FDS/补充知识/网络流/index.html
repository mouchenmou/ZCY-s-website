    最小割与最大流问题

    1. Flow-value lemma¶

    假设 \(f\) 是任意一个流，\((A, B)\) 是任意一个割。那么，跨越 \((A, B)\) 的净流量等于流 \(f\) 的值。

    • 流 \(f\) (Flow)：可以想象成水管网络中流动的水。流的值 \(f\) 就是整个网络从源点 \(s\) 真正发出的净水量。

    • 割 \((A, B)\) (Cut)：想象你在整个管网上切一刀，把所有节点分成两个互不相交的集合 \(A\) 和 \(B\)。通常要求起点 \(s\) 必须在集合 \(A\) 里，终点 \(t\) 必须在集合 \(B\) 里。

    • 净流量 (Net flow)：指的是从集合 \(A\) 流向集合 \(B\) 的总流量，减去从 \(B\) 反向流回 \(A\) 的总流量。

    • 核心结论：无论你怎么去“切”这个网络，只要把起点和终点分开了，穿过这个切面的净水量始终是不变的，并且严格等于整个网络总的流水量。

    这个其实想象一下很简单，想象出一个错综复杂的水管图，有一堆水管交错连接，随便将这堆水管切一刀。切完之后有些水是从B流回A的，有些是从A流向B的，后者减去前者就是总流量 \(f\)。

    2. Weak duality¶

    假设 \(f\) 是任意一个流，\((A, B)\) 是任意一个割。那么，流的值 \(\le\) 割的容量。

    这个定理的意思非常直观：无论你怎么切，实际流过去的水量，永远不可能超过这个切面所有水管的最大物理容量之和。

    2. 两个重要定理：¶

    2.1 Augmenting path theorem:¶

    A flow is a maxflow iff no augmenting paths.

    2.2 Maxflow-mincut theorem:¶

    Value of the maxflow = capacity of mincut.

    证明上面两个定理等价于证明下面三句话等价：

    • 条件 i: There exists a cut whose capacity equals the value of the flow \(f\).
    • 条件 ii: \(f\) is a maxflow.

    • 条件 iii: There is no augmenting path with respect to \(f\).

      • 在当前水流 \(f\) 的基础上，再也找不到任何一条增广路径了。

      • 增广路径就是一条从源头到终点，且一路上水管都没满的路径（只要没满，我们就能顺着这条路再多灌一点水）。找不到增广路径，意思就是滴水不进了，再也找不到任何可以偷偷加水的路线了。

    证明 \(i \implies ii\)：¶

    假设(A, B)是一个cut，并且这个cut的容量 = f

    那么，无论其它任何 \(f'\)，都会小于等于这个cut的容量（如果某个\(f'\) 超过了这个cut的容量，那么水管就爆炸了）。因此我们得出：

    \[ 任意f' \le f \]

    这也就说明了f是最大流，

    证明 \(ii \implies iii\)¶

    用反证法¶

    假设在flow的大小为 \(f\) 的情况下仍然存在增广路径，也就是说

    那么我们就可以通过这个增广路径来增加流量，即 \(Maxflow>f\)

    因此 \(\neg iii \implies \neg ii\)，得证。

    证明 \(iii\implies i\)¶

    假设 \(flow=f\) 时已经不存在增广路径了。

    1. 构造割 \((A, B)\)：
      • 定义集合 \(A\) 为：从源点 \(s\) 出发，通过无向路径（满足“正向边未满”或“反向边未空”）所能到达的所有顶点的集合。
      • 即路径上：没有满的正向边 (no full forward edges)，也没有空的反向边 (no empty backward edges)。

    2. 确定 \(s\) 和 \(t\) 的归属：

      • 根据定义，\(s\) 必然属于集合 \(A\)。

      • 因为假设不存在增广路径，说明按上述规则绝对无法到达汇点 \(t\)，因此 \(t\) 必然属于集合 \(B\)。

      • 分析割的容量与流量关系：
      • **从 \(A\) 指向 \(B\) 的前向边（Forward edges）：这些边必定是满载的。
        • 为什么？假设某条从 \(A\) 到 \(B\) 的边没有满，意味着它还有剩余容量。那么在残存网络中，从 \(A\) 里面的节点就可以顺着这条边走到 \(B\) 里面的节点。但这违背了我们对集合的定义！我们说过，\(B\) 里的节点是从 \(A\) 走不到的。所以，唯一的解释就是：这条边已经彻底塞满了，剩余容量为 0，实际流量 = 它的最大容量。

      • **从 \(B\) 指向 \(A\) 的后向边（Backward edges）：这些边必定是空载（流量为0）的。

        • 为什么？假设某条从 \(B\) 到 \(A\) 的边里有水流，那么Residual Graph中，就会有一条反向的边（后悔药），这条反向边是从 \(A\) 指向 \(B\) 的。如果这条反向边存在，意味着你可以从 \(A\) 走到 \(B\)，再次违背了集合定义。所以，唯一的解释就是：这条边里根本没有水，实际流量为 0。

    根据Flow-value lemma：\(穿过割的净流量=f。\)

    \(\because B->A的流量为0\)

    \(\therefore 净流量全都由A->B提供\)

    \(又\because A流向B的所有边都是满流。\)

    \(\therefore A->B的容量=A->B的流量\)

    \(又\because A->B的流量=净流量=f\)

    \(\therefore A->B的容量=f\)

    得证。

    寻找集合A的方法

    1. 第 1 步：从起点 \(s\) 出发
      • 尝试走到上方节点：实际流量 10，容量 10（10/10）。正向满载，路死，走不通。
      • 尝试走到中间节点：5/5。正向满载，走不通。
      • 尝试走到下方节点：13/15。没满！有 2 的余量。恭喜，我们成功走到了左下节点。把它拉进集合 \(A\)。
    2. 第 2 步：从“左下节点”继续探索
      • 尝试向右走到右下节点：16/16。正向满载，走不通。
      • 尝试向上走到左中节点：¾。没满！有 1 的余量。恭喜，我们成功“偷渡”到了左中节点。（你看，虽然从 \(s\) 直接过不去，但我们可以绕路过去）。把它拉进集合 \(A\)。
    3. 第 3 步：从“左中节点”继续探索
      • 尝试向右走到正中间节点：8/8。正向满载，走不通。
      • 尝试向上走到左上节点：注意，这里的箭头是从上往下的（0/4）。我们现在是逆着箭头走。规则说，逆向走水管不能是空的。但这根管子目前流量就是 0，没有后悔药。走不通。
      • 尝试向右下走到右下节点：仔细看图，有一根斜向左上的水管，是从右下指向左中的（6/6）。我们现在是从左中向右下走，也就是逆着箭头走。它空了吗？没空，里面整整有 6 吨水！有超大颗后悔药！恭喜，我们顺着这颗后悔药，成功走到了右下节点。把它拉进集合 \(A\)。
    4. 第 4 步：从“右下节点”继续探索
      • 向右走到终点 \(t\)：10/10。正向满载，走不通。
      • 逆向往上走：0/15。空的，没有后悔药，走不通。

    最终，集合 \(A\) 里面有 4 个节点：\(\{s, \text{左下}, \text{左中}, \text{右下}\}\)**。图上剩下的所有节点，全部归入集合 \(B\)。

    验证最大流 = 最小割¶

    1. 起点 \(s \to\) 左上节点：容量是10。
    2. 左中节点 \(\to\) 正中节点：容量是8。
    3. 右下节点 \(\to\) 终点 \(t\)：容量是10。

    最小割的容量= 10 + 8 + 10 =28。

    我们再回头看看源头 \(s\) 真正流出了多少水：

    3. Dinic 算法¶

    由于HW10中遇到了网络流的编程题，不得已去学了一下Dinic算法，但是能找到的视频都只是演示了Dinic算法的执行过程，根本不讲它的原理，所以我在这里想把原理也讲一下。

    可以先跟着这个视频走一走，熟悉一下Dinic算法的流程先。

    3.1 Dinic 到底想解决什么问题？¶

    Ford-Fulkerson 找增广路径的方式是盲目的。每次在残量图里随便找一条从 s 到 t 的路径，可能会找到一条很长很绕的路，走了好几步才灌了一丁点水。同样的边可能被反复经过，做了很多无用功。

    Dinic 的想法是：不要一条一条地找，我们一轮一轮地找。

    具体来说，Dinic 把找增广路径的过程分成了两个阶段：

    1. BFS 阶段：在残量图上跑一次 BFS，给每个节点标个层号（从 s 出发最短几步能到）
    2. DFS 阶段：在 BFS 标好的层号框架下找增广路径，只能从低层走向高层

    这两个阶段反复交替，直到找不到增广路径为止。

    3.2 分层是怎么回事？¶

    BFS 之后，每个节点都有一个层号：

    • \(level[s] = 0\)
    • \(level[v]\) = 从 s 到 v 在残量图中最短要走几步

    比如：

    1
2
3
4
5
6
7
             s (layer 0)
       /   |   \
      A    B    C     (layer 1)
       \  / \  /
        D    E        (layer 2)
         \  /
          t           (layer 3)

    分层之后有个非常关键的规则：只能从低层走向高层，也就是只能走 \(level[w] = level[v] + 1\) 的边。不可能走回头路，也不可能停在同一层。

    这意味着：在层次图里找到的任意一条 \(s \to t\) 路径，一定是当前残量图中 s 到 t 的最短路径。

    3.3 阻塞流¶

    Dinic 的 DFS 阶段不是找到一条增广路径就停下来，而是一直找，直到层次图里再也找不到 \(s \to t\) 的路径为止。

    什么意思呢？层次图里能走的路全被走断了，这条路堵了，那条路也堵了。这时候我们说找到了一个阻塞流。

    关键点： - 第 0 轮的残量图就是原图，没有任何流量。 - 每轮 DFS 灌完水之后, 残量图被更新了 (正向边容量减少，反向边容量增加）。 - 下一轮的 BFS 是在更新后的残量图上跑的，不是原来的图。

    3.5 为什么每轮之后 \(level[t]\) 会增加？¶

    一轮 DFS 把层次图里所有 \(s \to t\) 的路都堵死了。原来那些最短路径上，至少有一条边被灌满了。所以下次 BFS 再找，那些最短路径走不通了，只能走更长的路。这也是 Dinic 为什么快的根本原因

    执行流程：¶

    第一轮 BFS：假设 \(level[t] = 3\)。说明 s 到 t 最短要走 3 步。层次图里所有 \(s \to t\) 的路径长度都是 3。

    第一轮 DFS：在层次图里灌水，灌到再也灌不进去（阻塞流）。这意味着：层次图里每条 \(s \to t\) 的路径上，至少有一条边被灌满了。否则如果某条路径上所有边都还有余量，DFS 就能继续灌，不叫阻塞流。

    第二轮 BFS：重新在残量图上跑 BFS。原来那些长度为 3 的路径，因为至少有一条边满了，残量图里走不通了。

    会不会有一条长度还是 3 的路径，只是上一轮没被走到？

    不可能。如果残量图里还存在长度为 3 的 \(s \to t\) 路径，那上一轮 BFS 的时候这条路径就已经在层次图里了（BFS 会找到所有最短路径）。那为什么 DFS 没把它堵死？说明这条路径上所有边都还有余量。但这和"阻塞流"矛盾了——阻塞流的定义就是层次图里再也没有 \(s \to t\) 的路了。

    所以：下一轮如果还能走到 t，路径长度一定 > 3。即 \(level[t]\) 严格增加。

    问题2：那反向边（后悔药）会不会变出一条新的短路径？

    问题：下一轮BFS的时候，残量图里多出了反向边。会不会通过这些反向边，走出一条比原来还短的路径？

    这是不可能的。自行脑补吧，真的不知道该怎么解释了，我自己也在这里懵了很久还是放弃了。

    结论：

    • \(level[t]\) 每轮严格递增
    • 最多只有 \(|V|-1\)（因为最长路径不可能超过 \(|V|-1\) 条边）
    • 所以 Dinic 最多跑 \(|V|-1\) 轮 BFS

    3.6 当前弧优化¶

    Dinic 的代码里有一个 cur 数组，叫当前弧优化。

    什么意思呢？在同一个 DFS 阶段里（同一张层次图），如果从节点 \(v\) 出发的某条边已经被走到尽头了（这条路以及它后面的所有路都堵死了），那么下次 DFS 再回到 \(v\) 的时候，不需要再试这条边了。

    所以用 cur[v] 记录 \(v\) 当前试到第几条边了。DFS 回溯的时候，这个值不会被重置。这样每条边在一个 DFS 阶段里只会被检查一次。

    没有这个优化的话，Dinic 的时间复杂度会退化。

    // 全局变量
int level[MAXN];    // 每个节点的层号
int cur[MAXN];      // 当前弧优化：记录每个节点试到第几条边了

// BFS：构建层次图
bool bfs(int s, int t) {
    memset(level, -1, sizeof(level));
    level[s] = 0;
    queue<int> q;
    q.push(s);
    while (!q.empty()) {
        int v = q.front(); q.pop();
        for (each edge e from v) {
            if (e.cap - e.flow > 0 && level[e.to] == -1) {
                // 这条水管还有余量，而且 e.to 还没被标过层号
                level[e.to] = level[v] + 1;
                q.push(e.to);
            }
        }
    }
    return level[t] != -1;  // t 是否还能从 s 走到
}

// DFS：在层次图中找增广路径
int dfs(int v, int t, int pushed) {
    if (pushed == 0) return 0;
    if (v == t) return pushed;
    for (int &i = cur[v]; i < adj[v].size(); i++) {
        Edge &e = adj[v][i];
        if (level[v] + 1 != level[e.to] || e.cap - e.flow == 0)
            continue;  // 不是下一层或者水管满了，跳过
        int tr = dfs(e.to, t, min(pushed, e.cap - e.flow));
        if (tr == 0) continue;  // 这条路走不通
        e.flow += tr;            // 正向边加水
        adj[e.to][e.rev].flow -= tr;  // 反向边加水（后悔药）
        return tr;
    }
    return 0;
}

// Dinic 主函数
int dinic(int s, int t) {
    int flow = 0;
    while (bfs(s, t)) {       // 还能从 s 走到 t
        memset(cur, 0, sizeof(cur));  // 重置当前弧
        while (int pushed = dfs(s, t, INF))
            flow += pushed;     // 反复找增广路径
    }
    return flow;
}

    3.8 时间复杂度¶

    一般图的最坏情况：\(O(|V|^2 \cdot |E|)\)。推导如下：

    • BFS 轮数：每轮 \(level[t]\) 严格递增，最多增到 \(|V|-1\)，所以 BFS 最多跑 \(|V|-1\) 轮。每轮 BFS 本身是 \(O(|E|)\)。
    • 每轮 DFS：在当前弧优化下，每条边在一个 DFS 阶段里只被检查一次，所以一轮 DFS 的总工作量是 \(O(|V| \cdot |E|)\)。
    • 总复杂度：\((|V|-1)\) 轮 × \(O(|V| \cdot |E|)\) = \(O(|V|^2 \cdot |E|)\)。

    Tip

    在实际应用中，Dinic 的性能通常远好于理论最坏情况。对于大多数竞赛题目，Dinic 可以轻松处理 \(|V| \leq 1000, |E| \leq 5000\) 的数据规模。
