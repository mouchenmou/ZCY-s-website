

!!! abstract
    快速排序的核心操作是**分区（partitioning）**——选一个 pivot，把小于它的放左边、大于它的放右边。然后递归处理左右两半。最坏 $\Theta(N^2)$，但期望 $\Theta(N \log N)$，且实际中最快。

---

## 1. 故事背景

1960 年，Tony Hoare 在做俄语-英语翻译程序。字典存在磁带上（不能二分查找——磁带需要物理移动！），句子存在 RAM 里。

他的想法：**先把句子排序，然后扫一遍磁带查字典。** 排序花了 $N \log N$，查字典花了 $D$。

但问题来了——他用的语言**不支持递归**，所以他发明的这个递归排序算法在当时居然没法直接用。而且他译的句子通常不超过 15 个词，用快排纯属大炮打蚊子。但这个故事不重要——重要的是他发明的这个算法后来成了实际中**最快的排序**。

"Quicksort" 这个名字是 Tony 起的——幸运的是，它确实很快。

---

## 2. 分区（Partitioning）

快排的核心是一个叫 **partition** 的操作：选数组里某个元素 $x = a[i]$ 作为 **pivot**，重排数组使得：
- $x$ 移到位置 $j$（可能和 $i$ 相同）
- $x$ 左边所有元素 $\le x$
- $x$ 右边所有元素 $\ge x$

```
选 pivot = 5（最左边）
[5, 550, 10, 4, 10, 9, 330]
           ↓ partition
[4, 5, 9, 10, 10, 330, 550]
     ↑
   pivot 归位了
```

注意：partition 之后 **pivot 就已经在它最终排序好的位置上了**——左边全小于它，右边全大于它。这就是快排为什么行的关键！

分区本身的复杂度是 $\Theta(K)$，$K$ 是被分区的元素个数。

---

## 3. Quicksort 算法

```
Quicksort(N items)：
1. Partition on leftmost item（选最左元素做 pivot）
2. Quicksort left half（递归排左半）
3. Quicksort right half（递归排右半）
```

[完整演示看这里](https://docs.google.com/presentation/d/1SzcQC48OB9agStD0dFRgccU-tyjD6m3esrSC-GLxmNc/edit#slide=id.g463de7561_042)

```
原始：[32, 15, 2, 17, 19, 26, 41, 17, 17]
       ↓ partition(32)
      [15, 2, 17, 19, 26, 17, 17, 32, 41]
       ← ≤ 32 →              ↑    ← ≥ 32 →
                         pivot 归位
       ↓ 左右递归...
最终已排序！
```

## 4. 时间复杂度分析

复杂度完全取决于 **pivot 落在什么位置**。

### 4.1 最佳情况：pivot 每次都在正中间

```
              N              ← 每层总工作量 ≈ N
         N/2    N/2          ← ≈ N
      N/4 N/4 N/4 N/4        ← ≈ N
      ...                    ← 共 log N 层
```

每层工作量 $\approx N$，共 $\log N$ 层 → **$\Theta(N \log N)$。**

### 4.2 最坏情况：pivot 每次都在最边上

比如数组已经排好序 `[1, 2, 3, 4, 5, 6]`，选最左 = 1 做 pivot，它本来就在最左，partition 后原地不动。下一层剩 5 个，再下一层剩 4 个……

```
N → N-1 → N-2 → ... → 1
```

总工作量 = $N + (N-1) + \cdots + 1 = \Theta(N^2)$。

**所以如果数组已经排好序，选最左元素做 pivot 的 Quicksort 直接退化到 $\Theta(N^2)$。**

---

## 5. 那为什么实际中它最快？

### 论证一：只要 pivot 不太烂，还是 $O(N \log N)$

假设 pivot 每次都至少离边缘 10%（即落在 10%~90% 分位之间）。那每次递归，**较大那一半最多是 90%**。

$$N \to \frac{9}{10}N \to \left(\frac{9}{10}\right)^2 N \to \cdots$$

层数 $H \approx \log_{10/9} N = O(\log N)$。每层工作量 $O(N)$。

**就算 pivot 始终不在正中间，只要不总是在最边缘，快排还是 $O(N \log N)$！**

### 论证二：Quicksort = BST Sort

快排的比较过程和 BST 插入**完全一致**。比如数组 `[5, 3, 2, 1, 8, 4, 6, 7]`：

- pivot=5：所有数都和 5 比较一次
- 左半 pivot=3：1, 2, 4 只和 3, 5 比较
- 右半 pivot=7：6, 8 只和 7, 5 比较

这和把这组数随机插入 BST 的 `compareTo` 调用次数一模一样。而随机插入 BST 的期望复杂度是 $O(N \log N)$。

### 实证数据

对 $N=1000$，跑一万次试验：
- **平均比较次数**：$\approx 2N \ln N$
- 大多数在 ~12000 次比较
- 极少数 ~15000 次
- 达到 1000000 次的概率**几乎为零**

**结论：期望 $\Theta(N \log N)$，且常数因子比 Mergesort 小。**

---

## 6. 避免最坏情况

最坏 $\Theta(N^2)$ 在实际中确实会发生：
- **坏顺序**：已排序或接近排序的数组 → 每次都选到最小/最大 pivot
- **坏元素**：全是重复元素 → 分区不平衡

**解决方法：**

1. **用中位数做 pivot**：肯定好，但找中位数本身开销太大
2. **随机 shuffle**：排序前打乱数组。简单粗暴但**一定有效**——期望 $O(N \log N)$
3. **随机选 pivot**：不固定选最左，随机选一个。也能避免最坏

---

## 7. 总结

| 排序 | 内存 | 时间 | 备注 |
|------|------|------|------|
| Heapsort | $\Theta(1)$ | $\Theta(N \log N)$ | 缓存差 |
| Insertion | $\Theta(1)$ | $\Theta(N^2)$ | 几乎有序时 $\Theta(N)$ |
| Mergesort | $\Theta(N)$ | $\Theta(N \log N)$ | 稳定 |
| **Quicksort** | $\Theta(\log N)$ | $\Theta(N \log N)$ 期望 | **实际最快** |

空间 $\Theta(\log N)$ 来自递归调用栈（最坏可到 $\Theta(N)$，但期望 $\Theta(\log N)$）。
