# Git: A Command Line Tool

## Basic Git Function

### 1. 我们为什么要用版本控制？

1. 软件开发是一个迭代的过程
    - 写代码不是一蹴而就的，而是一遍遍修改、增加新功能、修复 Bug 的过程。
2. 我们保留多个副本是很有用的
    - 当与他人合作开发项目时，我们要确保不会破坏别人做好的工作。
    - 当独自一人工作时，我们有时也想进行一些试验性的修改。比如你想尝试重构一段代码，但怕改坏了原本能跑的程序，这时候你希望能有一个“安全存档”。


### 2. The Naive Approach
实现版本控制的Naive approach：把一堆旧版本存放在多个不同的文件夹里。例如：`2048终于能跑了` 文件夹，`2048修复了合并Bug` 文件夹等等：

- 优点：
    - 非常简单，只需要知道怎么复制文件就行
- 缺点：
    - 浪费存储空间
    - 需要非常小心，以避免错误的恢复和存储。比如手动管理太多文件容易搞混，一不小心就把错误的版本覆盖了正确的版本。
    - 需要很强的自律性才能做好备份，如果三天忘了复制备份，一旦出错可能就会丢失这三天的工作成果。
    - 合并两个产生歧义的副本需要非常仔细的手活。比如你写了功能A，队友写了功能B，我们得手动把这两份代码拼接到一起，解决冲突是很痛苦的。

### 3. Version Control Software
因为Naive Approach太笨了，所以有了专门的软件。市面上有许多处理版本控制的软件包，一些流行的系统包括：

- Git (2005, open source)
- Perforce (1995)
- SVN (2000, open source)
- Mercurial (2005, open source)

Git是最受欢迎的。

### 4. Git: How it works?
每次提交（`git commit -m "..."`）对文件的修改时，Git 都会在你的电脑上一个名为 `.git` 的隐藏文件夹中，存储整个项目的一个副本。

![](附件/Pasted%20image%2020260501164057.png)

1. `subl Hats.java` 和 `subl Cheese.java`：用文本编辑器（这里是 Sublime Text）创建并写了两个 Java 文件。
2. `git add . ; git commit -m "version 1 of my code"`：把这些文件交给 Git 管理，并拍下第一张“快照”（V1），备注是 "version 1 of my code"。
3. 接着，你修改了 `Cheese.java`，再次 `add` 和 `commit`，这就生成了第二张快照（V2），备注是 "fixed cheese bug"。

## Avoiding Redundancy
### 考虑以下场景：
假设一个程序员在开发项目时执行了三次`git commit`

- In V1: Create readme.txt
- In V2: Create utils/Utils.java, game/Game.java, game/Test.java. Modify readme
- In V3: Modify game/Game.java, change readme.txt back to V1 version

![](附件/Pasted%20image%2020260501175050.png)
### 方法一：Store Mutiple Copies of Everything
这就是刚才说的Naive approach的具体体现。下图的黑框展示了如果这么做，你电脑隐藏的 `.git` 文件夹里会是什么样子：每个版本都有一个专属文件夹，里面塞满了当时所有的文件。

![](附件/Pasted%20image%2020260501175302.png)

#### 优点：易于实现

- commit操作仅仅是创建一个子文件夹，然后把所有文件都复制进去
- 切换版本操作仅仅是把当前工作目录里的东西全删了，然后从你想要的那个备份文件里面把全部的文件复制进来并替换。

#### 缺点：效率低

- 2 个完全相同的 `Utils.java` 文件，在 V2 和 V3 里各存了一份
- 2 个完全相同的 `Test.java` 文件，在 V2 和 V3里各存了一份
- 2 个完全相同的 `readme.txt` 文件，在 V1 和 V3 里各存了一份（因为 V3 把它改回了 V1 的样子)

一个显而易见的改进方法：不要保存同一个文件的多个副本。

![](附件/Pasted%20image%2020260501180034.png)

### 方法二：Store only files that change
![](附件/Pasted%20image%2020260501180117.png)
在方法2中，每次 Commit 不再把所有文件复制一遍，而是只保存 **发生了变化的文件** 。对于没有变化的文件，直接用一个箭头（实际上就是“指针”或“引用”）指回它上一次被修改时所在的文件夹。

- **V3** 的节点，因为 `readme.txt` 改回了 V1 的样子，所以它的箭头直接越过 V2，指回了最初的那个蓝色的 `readme.txt`。
- `Utils.java` 没改过，所以 V3 的箭头指向 V2 里的 `Utils.java`。

#### 优点：效率高得多，避免了存储重复的文件

#### 缺点：Checkout变得更复杂了。

如果我们想签出一个旧版本，就必须从各种不同的文件夹里去拼凑复制文件。

![](附件/Pasted%20image%2020260501180946.png)

## Avoiding Redundancy with "Hashing"
哈希函数就像是一台 **数字指纹提取机：**

- **工作原理**：你把 **任何** 文件（不管是只有一行代码的 `.java` 文件，还是一部 10GB 的高清电影）扔进这台机器，它都会经过复杂的数学运算，吐出一串 **固定长度** 的乱码（比如 40 个字母和数字的组合）。这串乱码就是这个文件的哈希值（Hash），也就是它的 **数字指纹** 。
    
- **三大铁律**：
    1. **唯一性**：只要两个文件的内容 **一模一样** ，它们算出来的指纹就 **绝对一样** 。
    2. **极其敏感**：只要你改了文件里的 **哪怕一个标点符号** ，算出来的指纹就会 **面目全非** ，完全变成另一串字符。
    3. **不可逆**：你拿到指纹，绝对无法反向推导出文件原来的内容。

## 方法五：Use a "Hash" as the version number

Git 真正使用的方法，就是把你代码文件的 **内容** 放进名为 `git-SHA1` 的“指纹提取机”里。

- **The git-SHA1 hash is a deterministic function of the file's contents.**
    - 它是一个基于文件内容的确定性函数。（这就是我们刚才说的铁律 1：内容一样，指纹必然一样）。
    - `git-SHA1` 生成的指纹在计算机底层是 160 bits长。
        
- **Example (看右下角)**：
    - ![](附件/Pasted%20image%2020260501195236.png)
    
    - 左边这段极其简单的 `HelloWorld.java` 代码，扔进机器后，算出来的指纹就是右下角那串长长的十六进制代码：`66ccdc645c9d156d5c796dbe6ed768430c1562a2`。以后，这串字符就是这个文件内容的“身份证号”了！

### Using the git-SHA1 Hash（git是如何存储它的）
当你提交了刚才那个 `HelloWorld.java` 之后，Git 是怎么把它存在你的电脑里的呢？

1. 算出指纹：`66ccdc...`
2. **切割指纹建文件夹**：Git 会取这串指纹的 **前两个字符** (`66`），在隐藏的 `.git/objects/` 目录下建一个名为 `66` 的文件夹。
3. **存入文件**：然后把剩下的 38 个字符（`ccdc...`）作为**文件名**，把压缩后的代码内容存进去。
    - _(小提示：为什么要把前两个字符单独拿出来建文件夹？这是为了防止把几十万个文件全塞在同一个目录里导致电脑卡死，相当于做了一个简单的分类索引。)_

### SHA1-Hash的缺点

- 指纹只有 40 个字符（160 bits），容量虽然极大，但终究是有限的。而世界上可能写出来的代码文件是 **无限** 的。
- 根据抽屉原理（鸽巢原理），既然指纹数量有限，文件数量无限，那肯定存在 **两个内容完全不同的文件，刚好算出了同一串指纹** 。这就是“哈希碰撞（Collision）”。
- **Good news (好消息)**：
    - 发生碰撞的概率是 $1 / 2^{160}$。
    - 你需要制造出大概 $2^{80}$（也就是 $1.2 \times 10^{24}$，一万亿亿）个文件，才有可能碰到一次撞车。
- **结论**：理论上 Git 有这个 Bug，但在整个宇宙的生命周期里，这种事都不太可能发生。所以我们完全可以放心地把哈希值当做唯一的版本号。

### Add Benefit of SHA1-Hashing: Security（意外收获：极高的安全性）

- **Git uses the git-SHA1 hash to verify file integrity.**
    - Git 使用哈希值来验证文件的完整性。
- **场景假设**：
    - 你的项目里有一个极其重要的文件，Git 记录的它的指纹是 `ee380e...`。
    - 这时候有个黑客潜入了你的电脑，偷偷往这个文件里塞了一段 **恶意病毒代码**。
    - 但是黑客万万没想到，因为他改变了文件的内容（铁律 2 触发！），这个文件的新指纹瞬间就变了，再也不是 `ee380e...` 了。
    - 当你下次运行 Git 的时候，Git 拿现在的指纹和原来账本上的指纹一对比，立刻就会大声报警：“警告！指纹对不上，文件被篡改了！”。

**总结一下**：Git 放弃了用 `v1, v2` 这种笨方法，而是通过一套数学算法，为每一份代码的内容生成唯一的“数字指纹（哈希值）”。用指纹来管理文件，既不怕重名，又能防篡改，非常天才。

现在你理解了哈希是什么，你觉得在 Project 2 中，当你写代码创建一个新的 `Commit`（提交记录）时，你要如何利用哈希值把这个 Commit 和它里面包含的文件给关联起来呢？

### 我的疑问：
#### 疑问1：如果两个文件的哈希值都以 66 开头，放在同一个文件夹里不会冲突吗？

**答案是：会放在同一个文件夹里，但绝对不会冲突。这正是我们想要的结果！**

**详细解释：** 你把 `.git/objects` 想象成一个巨大的中药匣子柜。

- 柜子有 256 个抽屉（从 `00`, `01`, `02` 一直到 `ff`）。这也就是哈希值的前两位。
- 假设你有两个完全不同的文件：
    - 文件 A 的哈希值是：`66aaaaaa...`
    - 文件 B 的哈希值是：`66bbbbbb...`
- Git 会把这两个文件都扔进 `66` 这个抽屉（文件夹）里。
- 但是！在 `66` 文件夹里面，文件 A 的名字是 `aaaaaa...`，文件 B 的名字是 `bbbbbb...`。
- **因为它们的文件名完全不同，所以它们在同一个文件夹里平安无事，互不干扰。**
    

**为什么 Git 要费劲这么干？** 如果你的项目有 10 万个历史文件，你把 10 万个文件全塞在一个文件夹里，操作系统在查找文件时会非常慢（甚至卡死）。但如果你根据前两位分成 256 个文件夹，平均每个文件夹只有不到 400 个文件，电脑找起来就快如闪电了。这只是一种“负载均衡”的收纳技巧。

#### 疑问二：抛弃了 v1、v2，全变成了一堆哈希乱码，以后怎么拿出我想要的那个版本？

这是 Git 最精妙的设计。Git 并没有抛弃版本号，它只是 **把“找版本”和“存文件”这两件事分开了**。

**方法三 (带有清单的方法)正是破解疑惑的钥匙！

在 Git 里，不仅你的 **代码内容** 会被抽出哈希值存起来，你的 **每一次 Commit（提交动作）本身，也会变成一个文件被存起来！**

让我们来模拟一下真正的 Git 是怎么帮你找到 `v1` 的：

**第一步：给清单拍快照 (Commit Object)** 当你敲下 `git commit -m "version 1"` 时，Git 在底层偷偷创建了一个文本文件（这就是所谓的 Commit 对象）。这个文件里面的内容大概长这样：

- 作者：张三
- 时间：2025年5月6日
- 备注：version1
- 文件清单：
    - `HelloWorld.java` -> 去找哈希值 `66ccdc...`
    - `readme.txt` -> 去找哈希值 `8f14e...`
   

**第二步：清单也有自己的哈希值** Git 会把上面这个 Commit 文本文件，也塞进哈希提取机，算出另外一个全新的哈希值（比如 `a1b2c3...`）。这个 `a1b2c3...` 就是这个版本的真正“身份证号”。

**第三步：贴上人类看得懂的标签 (Branch/Tag)** 一长串 `a1b2c3...` 人类根本记不住。所以 Git 提供了一个叫 **分支 (Branch)** 或者 **标签 (Tag)** 的东西。 比如，你有一个叫 `master` 的分支。在 `.git` 文件夹里，就有一个极其简单的文本文件，名字叫 `master`，里面只写了一行字：

> `a1b2c3...`

#### 见证奇迹的时刻：Checkout 的逆向寻宝

当你输入命令 `git checkout master` 时，系统是这样顺藤摸瓜的：

1. Git 看到你要找 `master`，它就去看 `master` 文件，读到了 Commit 哈希值 `a1b2c3...`。
    
2. Git 跑进 `.git/objects/` 抽屉里，找到了 `a1b2c3...` 这个文件，把它打开。
    
3. Git 在里面看到了你当时留下的**清单**：“哦，在这个版本里，`HelloWorld.java` 对应的哈希值是 `66ccdc...`”。
    
4. Git 再次跑进 `.git/objects/66/` 文件夹，掏出名为 `ccdc...` 的文件，解压出来。
    
5. 把这个文件的内容扔进你的工作目录，重命名为 `HelloWorld.java`。

## Serializable（序列化） and Storing Data structures
Serializable（序列化）听起来是个非常高大上的专业术语，但它的核心概念其实就四个字：**“保存进度”**。

在解释下面的内容之前，需要先明白计算机里的一个基本常识：**内存（RAM） vs 硬盘（Disk）**：

- **内存**：你用 Java 写代码 `new` 出来的所有对象（比如一个 `Commit` 对象），都是活在内存里的。内存的特点是速度极快，但 **程序一结束，或者电脑一关机，里面的东西就全灰飞烟灭了**。
- **硬盘**：就是你的C盘、D盘，存进去的文件，只要你不删，它们就永远都在。

### Representing a Commit in Java
我们需要用Java的一个Class来表Commit，假设我们用下面的代码来表示Commit这个类：
```Java
    pubblic class Commit{
        public String author;
        public String date;
        public String commitMessage;
        public String parentID;
    }
```

- 假设我们的 Gitlet 程序中，Commit用了上面的代码所示的Class。在终端输入了 `java Gitlet commit "第一版"`。程序光速运行，在 **内存** 里 `new` 出了一个完美的 `Commit` 对象，包含了一切信息。
- **灾难来了**：Java 程序运行完这一行命令后，就 **结束退出** 了。程序一退出，内存被清空，你刚辛苦创建的 `Commit` 对象直接人间蒸发！等你下次输入 `java Gitlet log` 想查看提交记录时，程序去哪找刚才那个 Commit？它已经死了！
- **必须解决的问题**：我们必须想办法，在程序退出之前，把这个活生生的 Java 对象，**变成一个死的文件，永久保存在硬盘上（比如 `.git/objects/` 文件夹里）**。

为了解决这个问题，我们需要用到`Serializable`：

- **什么是序列化 (Serialization)？**
    - 简单来说，就是把一个在内存里立体的、活的 Java 对象，拍扁成一串长长的二进制字节流（01010101...），然后像写文本文件一样，把这串字节流写进硬盘里。
    - **反序列化 (Deserialization)** 就是它的逆过程：从硬盘里读出那串 0101，像泡面加水一样，把它在内存里重新复活成一个 Java 对象。
- **Java 贴心的内置功能**：
    - 把一个复杂的对象变成 0101 是一件极其复杂的麻烦事，但 Java 把这事儿全自动包办了。
    - 你只需要在你的 `class Commit` 后面加上一句咒语：`implements Serializable`。
        
- `implements Serializable`**这句咒语有什么用？**
    
    - PPT 里说 "There are no methods to implement (weird)"（不需要实现任何方法，有点奇怪）。没错，`Serializable` 被称为“标记接口” **。它就像是一张** 通行证。
    - 就是告诉 Java 虚拟机 : **我批准你把这个类的对象拍扁存进硬盘，你大胆去干吧！**
        
- **然后呢？**
    
    - 拿到通行证后，CS61B 的教授给提供了一个写好的 `Utils.java` 工具类。你只需要调用类似于 `Utils.writeObject(文件路径, 你的Commit对象)` 这样的方法，这个对象就会瞬间变成硬盘上的一个实体文件。
        

### 总结

因为你在写一个命令行程序，每次敲命令都是一次独立的程序运行，所以“把内存里的对象保存到硬盘上（序列化）”，以及“下次运行从硬盘读回内存（反序列化）”是贯穿整个 Project 2 最最核心的操作。

- 敲入`git checkout master`。
- 通过这行命令，把我们的视角（Git 里叫 `HEAD`，也就是“当前激活状态”）切回了主线任务。你的工作文件夹瞬间变成了 `7e` 那个时候的样子。
- 然后我们在主线上也写了点代码，提交生成了节点 **`27`**。
- **关键点**：现在两条时间线都在独立发展，互不干扰，形成了一个完美的 “Y” 字型分叉。

### 第三幕：世界线收束与冲突（Merging & Conflicts）

我们觉得`WithSwiss`分支上的`added swiss`功能写得挺好，想把它合入主线。

- 敲入 `git merge WithSwiss`（把`WithSwiss`这个分支合并到`Master`分支中）。
- **BOOM！冲突了 (CONFLICT)！**
- 为什么？因为在 `master` 分支和 `WithSwiss` 分支里，**都修改了同一个文件**（`Cheese.java`）。Git 是个聪明的机器人，但它不敢自作主张决定到底保留哪一边的代码，所以它“罢工”了。
- 右边那段带 `<<<<<<< HEAD` 和 `>>>>>>> Swiss` 的乱码，就是 Git 在你的代码里留下的“犯罪现场”。它在问你：“老板，上面这段是主线的，下面这段是分支的，你到底要哪段？你手动删减一下吧！”
![](附件/Pasted%20image%2020260501210243.png)
### 第四幕：解决冲突，合二为一 (Resolution)

#### 我们有三个选择：
比如说master中的`Cheese.java`实现了加法功能，`WithSwiss`分支实现了减法功能

1. 保留当前分支（master分支）的`Cheese.java`文件，也就是保留加法功能
2. 保留`WithSwiss`分支的`Cheese.java`文件，也就是保留剑法功能
3. 我都要，直接把`<<<<<`, `=====`, `>>>>>`全删掉，把加分功能和减法功能全都放进来

#### 冲突解决后：
- 这时候诞生了一个极其特殊的新节点：**`fa` (Merge Commit)**。
- 教授强调的重点 ：在合并之前，Commit 的历史记录是一条链子（Linked List），每个节点只有一个父节点。但在合并之后，节点 `fa` 拥有 **两个父节点**（`27` 和 `33`）！
- 这说明，Git 内部记录版本历史的数据结构，从一条线变成了一张 **网（Graph / 图）**。这就是你们这门课后面要学的核心数据结构。
