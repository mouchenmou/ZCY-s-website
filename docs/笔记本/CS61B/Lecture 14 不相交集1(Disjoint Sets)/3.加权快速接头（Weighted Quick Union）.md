## 1.准则：当我们将两棵树连接在一起时，应该始终将较小的树连接到较大的树下面。
## 原因：拿下面这个题目为例：
- 如果把小的树挂到大的树下面，那么大的数原有的那四个节点深度不变，因此查找速度不变，小的树深度+1，因此那两个节点的查找速度变慢。
- 如果把大的树挂到小的树下面，那么小的树原有的那两个节点深度不变，因此查找速度不变，大的树深度+1，因此那四个节点的查找速度变慢。

综合看待，肯定是把小的树挂到大的树下面更划算。

根据我们的准则，可以判定要选A选项

## 2.Possible Approach
One possible approach is to keep track of the height of every tree.(跟踪每棵树的高度)
- Link up shorter tree below the larger tree.
- In case of a tie, break tie arbitrarily.(就像系领带，可以随意打破领带)
Unfortunately, tracking tree height is kind of annoying.
Interestingly, tracking the tree's size, a.k.a. "weight" works just as well asymptotically(跟踪树的”大小“，也就是”权重“，同样也是渐进的)
- 大小和重量都表示该树中的项目总数。

### 我的证明方法（数学归纳法）
#### PS：PPT中有个向下取整，但是我没写出来
-  Base case：当 $N=1$ 的时，只有一个节点，$height=0$，$0 \  \le \ log_{2}N\ +\ 1$ 成立
- Inductive hypothesis：假设对所有$N \ \le \ k$ 的树 $T$，都满足$height(T) \ \le log_{2}N\ +\ 1$
- Inductive step：当 $N=k+1$ 的时候，这棵树可以看成是两棵树合并而成的，$T_1$ 的节点数为 $N_1$，$T_2$ 的节点数为 $N_2$，令 $N_1\ge N_2，\ N1+N2=k+1$，最坏的情况就是 $T_1$ 的树的高度是 $log_{2}N_1+1$，$T_2$ 的高度为 $log_{2}N_2+1$ ，$T$ 的高度$h \ \le max(height(T_1), \ height(T_2)+1)\ = max(log_{2}N_1+1, \ log_{2}N_2+2)$
    - 由于 $N_1<k+1$，所以 $log_{2}N_1+1\ \le log_{2}(k+1)+1$
    - 由于$N_2\  \le \frac{k+1}{2}$，所以 $log_{2}N_2+2 \ \le \ log_{2}(k+1)-1+2\ =\ log_{2}(k+1)+1$
    - 因此 $N=k+1$ 时 $height(T) \ \le \ log_{2}(k+1)+1$ 成立，假设得证。

## 5.Performance Summary
![](附件/Pasted%20image%2020251005191404.png)
QuickUnion's runtimes are O(H)，and weightedQuickUnionDS height is given by H=O(logN), therefore connect and isConnect are both O(logN).
通过调整quickuniond，我们实现了对数时间性能。快到足以解决所有实际问题。
因为isconnect的最坏情况是O(log N)，然而isconnect是由connect构建出来的，因此connect的最坏情况也是O(log N)。

## 一个小疑问：
有人问到加权重量查找是把元素少的树接到元素多的树下面，这种情况下connect和isConnect的时间都是O(logN)。但是有人会提问，为什么不把高度更矮的树接到高度更高的树下面去。当然这样也是可以的，并且这种方式的connect和isConnect也都是O(logN)，但是代码更难实现，因此我们还是把size更小的树接到size更大的树上。
