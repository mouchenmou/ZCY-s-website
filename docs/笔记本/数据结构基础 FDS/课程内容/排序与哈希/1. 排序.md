## 基本概念

### Stable

!!! conception "稳定排序"
    如果序列中有两个**相等**的元素 $A[i] = A[j]$（$i < j$），排序后它们的**相对位置不变**，则称该排序算法是**稳定的**。

!!! example "例子"
    原始：`(3, a), (1, b), (3, c)`，按数字排序。
    
    - 稳定排序结果：`(1, b), (3, a), (3, c)` — `(3, a)` 仍在 `(3, c)` 前面
    - 不稳定排序结果可能：`(1, b), (3, c), (3, a)` — 两个 3 的相对位置变了
    
---
### In-place

!!! conception "原地排序"
    排序过程中除了输入数组本身外，**只使用 $O(1)$ 的额外空间**（常数个临时变量）。

---

## 1. Insertion Sort

!!! abstract "类比：摸牌"
    就像打牌时摸牌——你手上已经有一把排好序的牌，摸到一张新牌后，从右往左找它应该在的位置，把比它大的牌都往右挪一位，然后把新牌塞进去。

```c
void InsertionSort(ElementType A[], int N) {
    int j, P;
    ElementType Tmp;

    for (P = 1; P < N; P++) {
        Tmp = A[P];                      /* 摸起下一张牌 */
        for (j = P; j > 0 && A[j - 1] > Tmp; j--)
            A[j] = A[j - 1];             /* 比 Tmp 大的牌往右挪 */
        A[j] = Tmp;                      /* 把新牌放到正确位置 */
    }
}
```

- 进行 $N-1$ 趟排序（第 $P$ 趟时，位置 $0$ 到 $P-1$ 已排好序）
- 第 $P$ 趟把第 $P$ 个元素插入到前面有序序列的正确位置

| 情况 | 复杂度 | 说明 |
|------|--------|------|
| 最坏 | $O(N^2)$ | 输入是**逆序**的，每趟都要把所有已排序元素右移 |
| 最好 | $O(N)$ | 输入已经**有序**，每趟只比较一次就结束 |
| 平均 | $O(N^2)$ | 见下方逆序对分析 |

### 1.1 逆序对（Inversion）与下界分析

!!! conception "逆序对（Inversion）"
    对于数组 $A$，若 $i < j$ 但 $A[i] > A[j]$，则称 $(i, j)$ 是一个**逆序对**。

!!! example "例子"
    数组 `34, 8, 64, 51, 32, 21` 有 **9** 个逆序对：
    
    (34,8) (34,32) (34,21) (64,51) (64,32) (64,21) (51,32) (51,21) (32,21)
    
    插入排序恰恰需要 9 次交换来把这个数组排好序。

**核心观察：** 插入排序每次交换两个相邻的元素，恰好消除恰好一个逆序对。

因此，插入排序的运行时间 = $O(I + N)$，其中 $I$ 是原数组中的逆序对数量。数组越接近有序（$I$ 越小），插入排序越快。

!!! abstract "两个定理"
    - **平均逆序对数量：** $N$ 个不同元素的数组中，平均有 $N(N-1)/4$ 个逆序对。
    - **下界定理：** 任何只交换相邻元素的排序算法，平均时间复杂度为 $\Omega(N^2)$。

!!! tip "为什么这很重要？"
    要想突破 $O(N^2)$，必须做相隔很远的元素之间的比较和交换——而不是只交换相邻元素。
    
    这就是希尔排序（Shell Sort）和后面那些 $O(N\log N)$ 算法的核心动机。

### 1.2 稳定性与原地性

- **Stable?** ✅ 稳定。`A[j-1] > Tmp` 只在严格大于时才移动，相等的元素不会穿越彼此。
- **In-place?** ✅ 原地。只用了一个临时变量 `Tmp`，额外空间 $O(1)$。

## 2. Shell Sort

!!! abstract "核心思想"
    插入排序每次只交换相邻元素，一次只消一个逆序对，太慢了。
    
    **希尔排序**：一次消除多个逆序对——先比较相隔很远的元素，逐渐缩小步长，最后 $h=1$ 退化回普通插入排序。

### 2.1 增量序列与过程

**增量序列** $h_1 < h_2 < \cdots < h_t$，其中 $h_1 = 1$。对每个 $h_k$，做一次 $h_k$**-sort**：把数组按间隔 $h_k$ 分组，每组内做插入排序。

!!! example "例子：Shell 增量序列（$h_t = \lfloor N/2 \rfloor, h_k = \lfloor h_{k+1}/2 \rfloor$）"
    ```
    原始: 81  94  11  96  12  35  17  95  28  58  41  75  15
    
    5-sort: 35  17  11  28  12  41  75  15  96  58  81  94  95
    3-sort: 28  12  11  35  15  41  58  17  94  75  81  96  95
    1-sort: 11  12  15  17  28  35  41  58  75  81  94  95  96
    ```

**重要性质：** $h_k$-sorted 的序列在 $h_{k-1}$-sorted 后仍保持 $h_k$-sorted。

### 2.2 代码实现

```c
void Shellsort(ElementType A[], int N) {
    int i, j, Increment;
    ElementType Tmp;
    for (Increment = N / 2; Increment > 0; Increment /= 2)
        for (i = Increment; i < N; i++) {
            Tmp = A[i];
            for (j = i; j >= Increment; j -= Increment)
                if (Tmp < A[j - Increment])
                    A[j] = A[j - Increment];
                else
                    break;
            A[j] = Tmp;
        }
}
```

### 2.3 复杂度分析

| 增量序列 | 最坏 | 平均 |
|----------|------|------|
| Shell（$h_t = \lfloor N/2\rfloor$） | $\Theta(N^2)$ | — |
| Hibbard（$h_k = 2^k-1$） | $\Theta(N^{3/2})$ | $O(N^{5/4})$ |
| Sedgewick（$1, 5, 19, 41, 109, \ldots$） | $O(N^{4/3})$ | $O(N^{7/6})$ |

!!! warning "Shell 增量序列的坏情况"
    当增量不互质时，后面的小增量可能几乎不起作用：
    ```
    1 9 2 10 3 11 4 12 5 13 6 14 7 15 8 16
    → 8-sort, 4-sort, 2-sort 后数组纹丝不动，只有 1-sort 干活
    → 退化到 $O(N^2)$
    ```
    **Hibbard 序列**解决了这个问题——连续增量没有公因子。

### 2.4 稳定性与原地性

- **Stable?** ❌ 不稳定。远距离交换时，相等元素可能被跳过，相对位置被打乱。
- **In-place?** ✅ 原地。额外空间 $O(1)$。

## 3. Heapsort

### 3.1 算法一：Naive Heapsort

Naive Heapsort并非不是stable的算法，需要用到辅助数组

```
1. BuildHeap(H);              // O(N)
2. for i = 0 to N-1:
       TmpH[i] = DeleteMin(H); // O(log N) each
3. for i = 0 to N-1:
       H[i] = TmpH[i];         // O(1) each
```

- 总时间：$O(N \log N)$
- **空间翻倍**：需要额外的 `TmpH[]` 数组

### 3.2 算法二：In-place Heapsort

改用**大顶堆**，堆顶（最大元素）换到末尾，然后下沉。下标 0 作为堆的起点。

```c
void Heapsort(ElementType A[], int N) {
    int i;
    for (i = N / 2; i >= 0; i--)   /* BuildHeap: O(N) */
        PercDown(A, i, N);
    for (i = N - 1; i > 0; i--) {
        Swap(&A[0], &A[i]);         /* DeleteMax */
        PercDown(A, 0, i);
    }
}
```

**复杂度：**
- 建堆：$O(N)$（从 N/2 下沉到 0）
- $N-1$ 次 DeleteMax，每次 $O(\log N)$
- 总时间：$O(N \log N)$

!!! abstract "定理"
    对 $N$ 个不同元素的随机排列做堆排序，平均比较次数为：
    $$2N\log N - O(N\log\log N)$$

!!! warning "实际表现"
    虽然堆排序平均时间最优，但实际中往往比 Sedgewick 增量序列的希尔排序**更慢**——缓存局部性差（元素到处跳）。

### 3.3 稳定性与原地性

- **Stable?** ❌ 不稳定。堆操作（下沉、交换）会打乱相等元素的相对顺序。
- **In-place?** ✅ 算法二是原地，额外空间 $O(1)$。算法一不是（空间翻倍）。

## 4. Mergesort

### 4.1 核心操作：合并两个有序列表

```
A: 1  13  24  26          B: 2  15  27  38
   ↑Aptr                     ↑Bptr

C: 1  2  ...              ← 每次比较 A[Aptr] 和 B[Bptr]，取小的放入 C
   ↑Cptr
```

合并两个总长为 $N$ 的有序列表时间为 $O(N)$。

### 4.2 递归实现

```c
void MSort(ElementType A[], ElementType TmpArray[],
                  int Left, int Right) {
    int Center;
    if (Left < Right) {
        Center = (Left + Right) / 2;
        MSort(A, TmpArray, Left, Center);         /* T(N/2) */
        MSort(A, TmpArray, Center + 1, Right);    /* T(N/2) */
        Merge(A, TmpArray, Left, Center + 1, Right); /* O(N) */
    }
}
```

```c
void Merge(ElementType A[], ElementType TmpArray[],
                  int Lpos, int Rpos, int RightEnd) {
    int LeftEnd = Rpos - 1;
    int TmpPos = Lpos;
    int NumElements = RightEnd - Lpos + 1;

    while (Lpos <= LeftEnd && Rpos <= RightEnd)
        if (A[Lpos] <= A[Rpos])
            TmpArray[TmpPos++] = A[Lpos++];
        else
            TmpArray[TmpPos++] = A[Rpos++];
    while (Lpos <= LeftEnd)   /* 复制左半剩余 */
        TmpArray[TmpPos++] = A[Lpos++];
    while (Rpos <= RightEnd)  /* 复制右半剩余 */
        TmpArray[TmpPos++] = A[Rpos++];
    for (int i = 0; i < NumElements; i++, RightEnd--)
        A[RightEnd] = TmpArray[RightEnd];  /* 拷回原数组 */
}
```

### 4.3 复杂度分析

$$
\begin{align*}
T(1) &= 1\\
T(N) &= 2T(\frac{N}{2}) + O(N)\\
&= 2^k T(\frac{N}{2^k}) + k \cdot O(N)\\
&= N \cdot T(1) + \log N \cdot O(N)\\
&= O(N + N\log N) = O(N\log N)
\end{align*}
$$

### 4.4 稳定性与原地性

- **Stable?** ✅ 稳定。Merge 时 `A[Lpos] <= A[Rpos]` 取左边，相等元素保持原始顺序。
- **In-place?** ❌ 不是原地。需要 $O(N)$ 的临时数组 `TmpArray[]`。虽然存在原地归并的算法，但非常复杂且常数大，实际不用。

!!! tip "归并排序的定位"
    归并排序需要线性额外内存且拷贝开销大，**很少用于内部排序**。但它对于**外部排序**（数据在磁盘上）非常有用。

## 5. Quick Sort

!!! abstract "地位"
    已知的**实际运行最快**的排序算法。
### 5.1 算法框架

```c
void Quicksort(ElementType A[], int N) {
    if (N < 2) return;
    pivot = 从 A[] 中选一个基准元素;
    将 S = A[] \ {pivot} 分成两个不相交集合：
        A1 = { a ∈ S | a ≤ pivot }
        A2 = { a ∈ S | a ≥ pivot };
    A = Quicksort(A1, N1) ∪ {pivot} ∪ Quicksort(A2, N2);
}
```

**核心操作：** 选 pivot → 分区（partition）→ 递归。**pivot 一旦放好位置，就不再移动。**

### 5.2 选取 Pivot

#### 方法一：pivot = A[0]（错误方法）

如果数组**恰好已经排好序**（比如 `{1, 2, 3, ..., N}`），每次选 `A[0]` 作为 pivot：

- 第一轮：pivot = 1，分区后左边为空，右边有 $N-1$ 个元素
- 第二轮：pivot = 2，分区后左边为空，右边有 $N-2$ 个元素
- ……
- 每一轮 pivot 都是当前子数组的**最小值**，分区极度不平衡

递推式：$T(N) = T(N-1) + O(N)$，展开就是 $O(N^2)$——快速排序退化成了"慢速排序"。

#### 方法二：pivot = random

随机选 pivot，避免有序输入的坏情况。但**随机数生成本身有开销**。

#### 方法三：三数中值法（Median-of-Three）✅

取 `A[left]`、`A[center]`、`A[right]` 三个数，**排序后取中间那个**作为 pivot。

!!! example "例子"
    `A[0] = 8, A[5] = 2, A[10] = 6` → 排序后 `2, 6, 8` → 取中间的 **6** 作为 pivot。

这样做的好处：pivot 绝不会是最大或最小的那个，即使原始数组已经有序，分区也不会出现一边空、一边全满的极端情况，避免了方法一的 $O(N^2)$ 退化。

### 5.3 Partitioning Strategy— 如何做到 In-place

了解了 Quick Sort 的思想之后，我们第一时间想到的实现方法基本就是：**开辟一个新数组**，把小于 pivot 的元素放左边，等于 pivot 的放中间，大于 pivot 的放右边。

```
输入数组：{82, 1, 4, 95, 0, 3, 59, 6, 28, 7, 69}, pivot = 59

新数组：[1, 4, 0, 3, 6, 28, 7, | 59 | 82, 95, 69]
         ← 小于 59 的放左边 →  pivot  ← 大于 59 的放右边 →
```

但这需要 $O(N)$ 的额外空间，不是 In-place。**我们应该如何操作才能在原数组上原地完成分区？**

### 解决方法：双指针扫描

#### **第一步：先把 pivot 换到最后面去。**

如下图，若我们选择6作为pivot，那么就将6放到数组末尾。
![](附件/Pasted%20image%2020260608221416.png)


#### **第二步：双指针扫描。**

设 `i` 从左往右扫，停在 ≥ pivot 的元素上；`j` 从右往左扫，停在 ≤ pivot 的元素上。两者都停下后交换 `A[i]` 和 `A[j]`，继续扫描。当 `i` 和 `j` 交错时，分区完成。

如下图，$i=0$ 时 $A[i]=8>6$，因此 $i$ 直接停下。

$j=8$ 时$A[j]=7>6$，$j$ 接着移动到 $7$，$j=7$ 时 $a[j]=2<6$，因此j停下，此时 $i$ 与 $j$ 都已停下，此时便得到了下图所示的情景：
![](附件/Pasted%20image%2020260608221632.png)

接下来要做的是交换一下 $A[i]$ 和 $A[j]$，然后接着走，一直走到 $i$ 与 $j$ 交错为止。下图就是最终 $i$ 与 $j$ 最终的位置:

![](附件/Pasted%20image%2020260609002000.png)

#### **第三步：把 pivot 放回正确位置。
将 pivot 与 `A[i]` 交换——此时 `i` 正好是 pivot 该在的位置。
![](附件/Pasted%20image%2020260608224949.png)

**时间复杂度：** $O(N)$（每个元素恰好被访问一次）。

#### 当元素等于 Pivot 时该怎么办？

这是一个关键的实现决策。假设扫描时遇到一个值**恰好等于 pivot**：

##### 方案一：i 停，j 也停（正确做法）

考虑一个极端情况：**数组里所有元素都相等**（比如全是 5）。

- i 和 j 各走一步就都停下来
- 交换两个 5（看起来是"浪费时间"的交换）
- 然后继续，i 和 j 交替前进，在中点附近相遇
- 分区结果：**左右基本各一半** → 均衡
- 递推：$T(N) = 2T(N/2) + O(N) = O(N\log N)$

##### 方案二：i 不停，j 也不停

同样全是 5 的数组：

- i 一路往右冲到末尾（每个元素都"≥ pivot"，i 都不停）
- j 一路往左冲到开头（每个元素都"≤ pivot"，j 都不停）
- **一次交换都没发生**，i 和 j 直接擦肩而过
- 最终 pivot 被放到了数组的**最后一个位置**（因为i冲到了最右边，i停下来的位置就是pivot的位置）
- 每一轮递归只减少一个元素（只有 pivot 被"排好"），其他 $N-1$ 个元素进入下一层
- $T(N) = T(N-1) + O(N) = O(N^2)$

!!! tip "结论"
    方案一虽然在全等数组上做了一堆"看起来没用"的交换，但这些交换恰好让等于 pivot 的元素被均匀分散到两边，保证 $O(N\log N)$。PPT 原话："stop both i and j, and take some extra swaps."

### 5.4 小数组 cutoff

对于小 $N$（$N \le 20$），快速排序比插入排序**慢**——因为递归开销大于直接插入的代价。

**解决：** 当子数组长度小于某个阈值（如 10），改用插入排序。

### 5.5 完整实现（Median3 + Qsort）

```c
/* 三数中值法：取 left, center, right 的中位数作为 pivot */
ElementType Median3(ElementType A[], int Left, int Right) {
    int Center = (Left + Right) / 2;
    if (A[Left] > A[Center])   Swap(&A[Left], &A[Center]);
    if (A[Left] > A[Right])    Swap(&A[Left], &A[Right]);
    if (A[Center] > A[Right])  Swap(&A[Center], &A[Right]);
    /* 此时 A[Left] ≤ A[Center] ≤ A[Right] */
    Swap(&A[Center], &A[Right - 1]); /* 把 pivot 藏到 Right-1 */
    return A[Right - 1];             /* 返回 pivot */
}

void Qsort(ElementType A[], int Left, int Right) {
    int i, j;
    ElementType Pivot;
    if (Left + Cutoff <= Right) {
        Pivot = Median3(A, Left, Right);
        i = Left;  j = Right - 1;
        for ( ; ; ) {
            while (A[++i] < Pivot) { }  /* 从左扫描 */
            while (A[--j] > Pivot) { }  /* 从右扫描 */
            if (i < j)
                Swap(&A[i], &A[j]);     /* 交换错位元素 */
            else
                break;                   /* 分区完成 */
        }
        Swap(&A[i], &A[Right - 1]);     /* 恢复 pivot */
        Qsort(A, Left, i - 1);          /* 递归左半 */
        Qsort(A, i + 1, Right);         /* 递归右半 */
    } else
        InsertionSort(A + Left, Right - Left + 1); /* 小数组用插入排序 */
}
```

!!! tip "为什么 pivot 要藏到 Right-1 而不是 Right？"
    三数排序之后：`A[Left] ≤ A[Center] ≤ A[Right]`。这意味着：
    
    - `A[Left]` 已经 ≤ pivot，也不需要再跟pivot进行比较了
    - `A[Right]` 已经 ≥ pivot，也不需要再跟pivot进行比较了
    
    所以真正需要分区的范围是 `A[Left+1]` 到 `A[Right-1]`。我们把 pivot（即 `A[Center]`）和 `A[Right-1]` 交换——让 pivot 待在分区的边界上，等分区完成后一步放回正确位置。
    
    这也解释了为什么 `i` 的初值设为 `Left` 而不是 `Left+1`：因为用的是 **`++i`（前置递增）**——第一次循环 `i` 先变成 `Left+1`，再和 pivot 比较，所以 `A[Left]` 确实被跳过了。同理 `j` 初值设为 `Right-1`，`--j` 让第一次比较落在 `Right-2`。左右各省了一次。

### 5.6 时间复杂度分析

从上面的代码可以看出，Quick Sort 的时间由两部分组成：分区（$O(N)$） + 对左右两半递归排序。

设 pivot 最终落在位置 $i$（第 $i$ 个元素被排好），左半有 $i$ 个元素，右半有 $N-i-1$ 个元素，分区本身花 $cN$ 时间。因此**一般递推式**为：

$$T(N) = T(i) + T(N-i-1) + cN$$

下面这个式子的三种特殊情况对应三种复杂度：

#### 最坏情况：$i = 0$ 或 $i = N-1$

pivot 每次都是最小或最大的元素，分区后一边为空，另一边剩 $N-1$ 个。代入 $i=0$：

$$T(N) = T(N-1) + cN \;\Rightarrow\; O(N^2)$$

典型触发条件：数组已经排好序 + pivot 取 `A[0]`。

#### 最好情况：$i \approx N/2$

pivot 恰好是中位数，左右基本对半分。代入 $i = N/2$：

$$T(N) = 2T(N/2) + cN \;\Rightarrow\; O(N\log N)$$

三数中值法让这个情况在大多数输入下都能实现。

#### 平均情况：

假设 pivot 落在 $0 \sim N-1$ 任意位置的概率均等，则 $T(i)$ 的期望取平均，得到：

$$T(N) = \frac{2}{N}\sum_{j=0}^{N-1}T(j) + cN$$

**推导过程：**

1. 两边同乘 $N$：

$$N \cdot T(N) = 2\sum_{j=0}^{N-1}T(j) + cN^2$$

2. 写出 $N-1$ 的版本：

$$(N-1) \cdot T(N-1) = 2\sum_{j=0}^{N-2}T(j) + c(N-1)^2$$

3. ① − ②，消去求和：

$$N \cdot T(N) - (N-1) \cdot T(N-1) = 2T(N-1) + c(2N-1)$$

4. 整理：

$$N \cdot T(N) = (N+1) \cdot T(N-1) + c(2N-1)$$

5. 两边同除以 $N(N+1)$：

$$\frac{T(N)}{N+1} = \frac{T(N-1)}{N} + \frac{c(2N-1)}{N(N+1)}$$

6. 令 $S(N) = \dfrac{T(N)}{N+1}$，则 $S(N) = S(N-1) + O(1/N)$，累加得 $S(N) = O(\log N)$。

7. 回代：$T(N) = (N+1) \cdot S(N) = O(N\log N)$。

核心结论：虽然最坏是 $O(N^2)$，但平均性能和最好情况一样是 $O(N\log N)$，加上常数因子极小，快速排序是实际中跑得最快的比较排序。

### 5.7 稳定性与原地性

- **Stable?** ❌ 不稳定。分区时远距离交换破坏相对顺序。
- **In-place?** ✅ 原地（递归栈 $O(\log N)$ 另算）。

## 6. 排序问题的下界

!!! abstract "定理"
    任何只通过**比较**来决定顺序的排序算法，最坏时间复杂度为 $\Omega(N\log N)$。

**证明（决策树）：**

![](附件/Pasted%20image%2020260609002419.png)

- 排序 $N$ 个不同元素，有 $N!$ 种可能的结果
- 决策树的每一片叶子对应一种排序结果 
- 高度为 $k$ 的二叉树最多有 $2^{k-1}$ 片叶子

$$N! \le 2^{k-1} \quad\Rightarrow\quad k \ge \log(N!) + 1$$

由于 $N! \ge (N/2)^{N/2}$：

$$\log(N!) \ge \frac{N}{2}\log\left(\frac{N}{2}\right) = \Theta(N\log N)$$

所以 $T(N) = k \ge c \cdot N\log N$。

!!! tip "这意味着什么？"
    归并排序和堆排序的 $O(N\log N)$ 已经**触及了比较排序的理论下界**——不可能有比较排序比它们更快（在渐进意义上）。桶排序和基数排序之所以能 $O(N)$，是因为它们**不靠比较**。

## 7. Table Sort（间接排序）


如果排序的元素是**大型结构体**，每次交换都要拷贝整个结构体，非常昂贵。

**解决方案：** 不交换结构体本身，而是维护一个**指针表（table）**
![](附件/Pasted%20image%2020260609002057.png)
图中的`a, b, c, d`等等都是结构体，比大小的时候我们不去改变这些key的位置，而是去改变table的下标：

![](Pasted%20image%2020260609000638.png)
最终的`sorted list`就是`list[table[0], table[1], table[2] ..., table[n-1]]`，此处的顺序就是`list[4], list[1], list[3], list[0], list[5],  list[2]`

#### **最终物理重排：**

##### **第一步：从索引 0 开始，拉出一个环**

1. 我们准备处理索引 `0`。首先看 `table[0]`，它的值是 `4`。这说明现在 0 号位置的 `d` 是占着茅坑的，真正的“主人”应该是索引 4 的元素。
2. 为了腾出位置，我们把 0 号位置的 `d` 拿出来，暂存到一个临时变量里：`Temp = d`。现在，**索引 0 变成了空槽**。
3. 既然索引 0 空了，我们就去把它的主人请过来。根据 `table[0] = 4`，我们把索引 4 的 `a` 搬到索引 0。
    - _此时 key 数组变成：`[a, b, f, c, 空, e]`_
4. 现在，**索引 4 变成了空槽**。谁该来填这个坑？我们去看 `table[4]`，它的值是 `5`。所以把索引 5 的 `e` 搬过来。
    - _此时 key 数组变成：`[a, b, f, c, e, 空]`_
5. 现在，**索引 5 变成了空槽**。谁该来填？看 `table[5]`，值是 `2`。把索引 2 的 `f` 搬过来。
    - _此时 key 数组变成：`[a, b, 空, c, e, f]`_
6. 现在，**索引 2 变成了空槽**。谁该来填？看 `table[2]`，值是 `3`。把索引 3 的 `c` 搬过来。
    - _此时 key 数组变成：`[a, b, c, 空, e, f]`_
7. 现在，**索引 3 变成了空槽**。谁该来填？看 `table[3]`，值是 `0`。
    - **注意！** 出现 `0` 了！这意味着下一个该搬进来的是最初占据索引 0 的那个元素。我们形成了一个**闭环**！
    - 此时，我们把一开始保存在 `Temp` 里的 `d`，填入这个空槽（索引 3）。
    - _此时 key 数组变成：`[a, b, c, d, e, f]`_

这就是一个完整的“环”的重排。在这个环里走过的数据（a, e, f, c, d），一步到位地回到了它们最终的物理位置上。

![](附件/Pasted%20image%2020260609005629.png)
##### **第二步：检查剩下的索引**

- 接下来检查索引 `1`。看 `table[1]`，值是 `1`。这说明 `b` 本来就在正确的位置上，不需要移动。
- 再往后检查索引 2、3、4、5 时，算法会发现它们在第一步的那个大环里已经被处理过了（在代码实现中，通常会将处理过的 `table[i]` 标记一下，比如改写为 $i$ 本身，避免重复移动）。
    

!!! question "搬结构体不还是有拷贝开销吗？"
    搬结构体本身确实要拷贝。但关键区别在于：
    
    - **不用 table sort**：每一轮排序的比较和交换都要拷贝整个结构体。$O(N\log N)$ 次结构体级别的 swap，每次 $m$ 字节。
    - **用 table sort**：排序阶段只操作 table（整数数组），交换的是 4 字节整数。$O(N\log N)$ 次**整数**交换，极快。最后的物理重排最多 $\lfloor 3N/2 \rfloor$ 次结构体移动，是 $O(N)$ 而不是 $O(N\log N)$。
    
    而且物理重排是**可选的**——如果不需要 `list[]` 真正排好，只要能按顺序访问就行，那直接按 table 的顺序读就可以了，一步都不搬。

## 8. Bucket Sort（桶排序）

!!! example "场景：N 个学生按成绩（0-100）排序"
    成绩范围 $M=101$ 已知且不大。不需要比较——直接用成绩做下标，每个成绩一个桶。

```
count[0..100] 初始化为 0
while (读入一个学生的 record):
    count[stdnt.grade].append(record)   // 放入对应成绩的桶
for i = 0 to 100:
    if (count[i] 非空)
        按序输出 count[i] 中的所有记录
```

![](附件/Pasted%20image%2020260609003213.png)

成绩 88 的学生全部落入第 88 号桶，成绩 100 的落入第 100 号桶——一次遍历输入、一次遍历桶就排好了。**和输入顺序无关，完全绕开了比较。**

**时间复杂度：** $O(N + M)$。如果 $M = O(N)$（比如 $N$ 个学生成绩范围也是 $O(N)$ 量级），就是线性时间。

!!! warning "限制"
    当 $M \gg N$ 时（比如 $N=100$ 个学生，成绩范围却是 $0\sim 99999$），绝大部分桶是空的——遍历空桶浪费大量时间。此时用基数排序。

## 9. Radix Sort（基数排序）

!!! question "动机"
    桶排序在 $M \gg N$ 时效率差。但如果我们把 key **拆成多段**，每段的范围都不大，就能对每段分别桶排序。这就是基数排序。

### 9.1 LSD 基数排序

**从最低位（Least Significant Digit）开始**，依次向高位进行。每一趟是一个桶排序，但只针对**当前位**。

!!! example "例子：10 个整数范围 0-999（$M=1000$，太大）"
    1. 第一轮：按个位数进行分桶
    2. 第二轮：按十位数进行分桶
    3. 第三轮：按百位数进行分桶
    
    我们会发现，进行低2轮的时候，每个桶中的数都已经按照个位数排好序了，比如第二轮的0号桶中有`0, 1, 8`， 1豪桶中有`512, 216`等等，它们的个位数都是已经排好序的。
    
    同理，进行第三轮也是一样，只是这个例子中的数太少了，没有发生第三轮中一个桶中有两个数。
    
    ![](附件/Pasted%20image%2020260609003509.png)

合并：`0, 1, 8, 27, 64, 125, 216, 343, 512, 729` ✅

**时间复杂度：** $O(P(N + B))$，$P$ = 位数（pass 数），$B$ = 基数（每趟的桶数，如上例中 $B=10$）。

### 9.2 多关键字排序

把每个记录 $R_i$ 看成有 $r$ 个 key：$K_i^0$（最高位）到 $K_i^{r-1}$（最低位）。**字典序**定义为：从最高位开始逐位比较，碰到第一对不同 key 时决定大小。

!!! example "扑克牌就是 2 个 key"
    - $K^0$ = 花色：$\clubsuit < \diamondsuit < \heartsuit < \spadesuit$
    - $K^1$ = 点数：$2 < 3 < \cdots < A$

### 9.3 MSD vs LSD

| | MSD（最高位优先） | LSD（最低位优先） |
|---|---|---|
| 做法 | 按 $K^0$ 分 4 桶 → 每桶内递归排序 | 按 $K^1$ 分 13 桶 → 合并 → 按 $K^0$ 分 4 桶 |
| 特点 | 自然产生字典序；每桶独立，可并行 | 每趟必须使用**稳定**的排序（桶排序本身是稳定的） |
| 开销 | 递归、桶数不固定 | 迭代、每趟桶数一样 |

!!! example "扑克牌 MSD"
    先按花色分 4 堆 → $\clubsuit$ 堆里有 3 张 $\clubsuit$ 牌，堆内再按点数排 → 每堆独立排序，合并即得全序。

!!! example "扑克牌 LSD"
    先按点数分 13 堆 → 合并 → 再按花色分 4 堆 → 合并即得全序。注意：第一趟（按点数）必须稳定，否则合并后同花色的点数顺序会乱。

### 9.4 稳定性与原地性

- **Stable?** ✅ 稳定（前提是每一趟使用的排序算法本身稳定）。
- **In-place?** ❌ 不是原地，需要桶数组。

## 10. 各算法 Stable 与 In-place 总结

| 算法             | Stable | In-place | 备注                           |
| -------------- | ------ | -------- | ---------------------------- |
| Insertion Sort | ✅      | ✅        | 严格大于才移动，不破坏相等顺序              |
| Shell Sort     | ❌      | ✅        | 远距离跳跃打乱相对顺序                  |
| Heapsort       | ❌      | ✅（算法二）   | 堆操作打乱相对顺序                    |
| Mergesort      | ✅      | ❌        | `<=` 取左边保证稳定，但需要 $O(N)$ 辅助空间 |
| Quick Sort     | ❌      | ✅        | partition 的远距离交换打乱顺序         |
| 基数排序           | ✅      | ❌        | 取决于每位的排序是否稳定，需要桶             |
